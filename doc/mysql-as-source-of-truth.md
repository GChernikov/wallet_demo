# MySQL как источник истины для финансовых событий кошельков

## Контекст

Сейчас источником истины является **NATS JetStream**: все события кошелька пишутся в стрим с CAS-семантикой, балансы вычисляются из событий, проекции кэшируются в Redis. Этот документ оценивает **техническую возможность перехода на MySQL** как источник истины для балансов. Конкретная схема и план миграции — отдельный этап.

---

## Что проверялось

Две стратегии конкурентного обновления баланса под нагрузкой **3 000 RPS**:

- **SELECT FOR UPDATE (SFU)** — пессимистичная блокировка строки на всё время транзакции
- **Optimistic Locking (OL)** — версионный CAS, при конфликте retry до 5 раз, иначе HTTP 409

Два сценария нагрузки:
- **Низкий contention** — 750 кошельков × 4 RPS
- **Высокий contention** — 100 кошельков × 30 RPS

Три среды:

| Среда | MySQL | Durability | fsync cost |
|-------|-------|-----------|------------|
| Docker (benchmark) | 8.4, Docker Desktop | `flush=2, sync=0` (unsafe) | ~0 |
| Docker (durable) | 8.4, Docker Desktop | `flush=1, sync=1` (full ACID) | ~5–10 ms |
| Local native | 9.6, Homebrew macOS | OS defaults | ~0.1–0.5 ms* |

*macOS `fsync()` не гарантирует физический flush на диск — результаты оптимистичнее, чем на Linux bare metal.

---

## Результаты — 3 000 RPS

### Низкий contention (750 кошельков × 4 RPS)

| Стратегия | Docker benchmark p95 | Docker durable p95 | Local p95 |
|-----------|--------------------|--------------------|-----------|
| SFU | 244 ms ✓ | 171 ms ✓ | 139 ms ✓ |
| OL | 32 ms ✓ | 180 ms ✓ | 1.3 ms ✓ |
| Ошибки | 0% / 0% | 0% / 0.01% | 0% / 0% |

Все три среды справляются без ошибок. При durable-настройках в Docker p95 ~170–180 ms — но это Docker Desktop с виртуализированным fsync (~5–10 ms), а не реальное железо. Для оценки production-latency нужен тест на выделенных стендах с NVMe SSD.

---

### Высокий contention (100 кошельков × 30 RPS)

| Стратегия | Docker benchmark p95 | Docker durable p95 | Local p95 |
|-----------|--------------------|--------------------|-----------|
| SFU | 68 ms ✓ | **2 860 ms ❌** | 25.8 ms ✓ |
| OL | 20 ms ✓ | **904 ms, 8.7% ошибок ❌** | 1 ms ✓ |

Docker durable ломается при высоком contention. Local native справляется чисто.

## Ключевые выводы

### 1. MySQL выдерживает 3 000 RPS — при правильном размещении

При co-location (приложение и БД на одном хосте или в одном rack):
- 3 000 RPS обрабатываются без ошибок и с приемлемой latency
- Узкое место — не lock contention, а **fsync latency**: каждый COMMIT ждёт диск

### 2. Docker с durable-настройками — не репрезентативная среда для нагрузки

Docker Desktop эмулирует fsync через VirtioFS/QEMU (~5–10 ms), что недостижимо для production NVMe SSD (~0.1–0.2 ms). На реальном железе разрыв между benchmark и durable режимами будет в 25–50 раз меньше. Тест в Docker с `flush=1, sync=1` намеренно показывает **худший сценарий**, а не типичный production.

### 3. При умеренной нагрузке на кошелёк оба метода блокировки работают

При ≤ 4–5 RPS на один кошелёк (низкий contention):
- SFU и OL оба дают 0% ошибок
- OL быстрее по latency (нет lock wait), но разница несущественна при fast storage

При ≥ 30 RPS на один кошелёк (высокий contention) поведение расходится:
- **SFU**: 0% ошибок, latency растёт пропорционально очереди. Деградирует медленно, предсказуемо
- **OL**: конфликты исчерпывают retry budget → HTTP 409. При перегрузке доля ошибок растёт нелинейно (47% при 10K RPS в перегруженной системе)

Для финансового сервиса **SFU предпочтительнее**: клиент всегда получает 200 или инфраструктурную ошибку, никогда — "попробуй снова".

### 4. Idempotency и outbox — уже в схеме

Переход с NATS потребует гарантий:

| Требование | Решение в текущей схеме                                                     |
|-----------|-----------------------------------------------------------------------------|
| Ровно одно применение события | Таблица `idempotency_keys` (CAS через `ON DUPLICATE KEY UPDATE`)            |
| Надёжная доставка downstream-событий | Таблица `event_outbox` (transactional outbox pattern)                       |
| История транзакций кошелька | Таблица `transactions`, PK `(wallet_uuid, created_at, id)` — clustered scan |

Outbox-паттерн в MySQL заменяет роль NATS-стрима как шины событий для downstream-потребителей — события не теряются при краше, т.к. пишутся атомарно с балансом в одной транзакции.

### 5. Что нужно для production

- **Co-location**: приложение и MySQL в одном ЦОД/стойке, RTT < 1 ms
- **NVMe SSD**: fsync ~0.1–0.2 ms — тогда durable-режим сравним с Docker benchmark-режимом
- **MySQL-инстанс рядом с сервисом**: не shared dev-хост

При этих условиях 3 000 RPS на одном MySQL-инстансе — реалистичная цель. Для окончательного подтверждения нужен нагрузочный тест на выделенном стенде с durable-настройками на Linux + NVMe.

---

## Вывод

MySQL **способен** обслуживать 3 000 RPS для конкурентных обновлений балансов при условии co-location и fast storage. Существующие паттерны (idempotency, transactional outbox) закрывают ключевые требования к надёжности. Открытый вопрос — подтверждение latency на выделенных стендах с Linux + NVMe. Стратегию блокировки, схему и план перехода — отдельным этапом.
